Overview

Security Groups act as instance-level virtual firewalls. This guide explains inbound vs outbound flows, differences between Security Groups and Network ACLs, and includes a production-ready Terraform example for a three-tier architecture.

Inbound & Outbound Traffic Flow

Diagram: how security group rules are evaluated for inbound and outbound packets.

Security Groups vs Network ACLs

Security Groups: instance-level and stateful. Network ACLs: subnet-level and stateless. Use NACLs for coarse filtering at subnet boundary and SGs for instance-specific controls.

Terraform Example (three-tier)

resource "aws_security_group" "web_tier" {
  name        = "web-tier-sg"
  description = "Security group for web tier - allows public HTTP/HTTPS"
  vpc_id      = var.vpc_id

  ingress {
    description = "HTTP from public internet"
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }

  ingress {
    description = "HTTPS from public internet"
    from_port   = 443
    to_port     = 443
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }

  ingress {
    description     = "SSH from bastion host"
    from_port       = 22
    to_port         = 22
    protocol        = "tcp"
    security_groups = [aws_security_group.bastion.id]
  }

  egress {
    description = "Allow all outbound"
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }
}

Key Takeaways

  • Use least privilege ‚Äî only open necessary ports and sources.
  • Prefer SSM Session Manager over opening SSH to internet.
  • Use prefix lists for AWS service endpoints when possible.
